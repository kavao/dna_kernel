## 目的

課金・API 呼び出し・ファイルの大量生成・外部サービスへの送信など、
**やり直しが効かない・コストが発生する処理**に対して、
計画確認と明示承認を挟む構造を固定する。

## 基本フロー（4ステップ）

```
1. 設定確認  →  2. dry-run 提示  →  3. ユーザー承認  →  4. 本番実行 + 完了確認
```

### ステップ1: 設定確認

実行前に、provider・API キー・設定ファイルを確認する。
**設定を確認せずに dry-run から始めない。**

```
確認項目の例:
  - API キーが .env に存在するか
  - 設定ファイルの provider・モデル・保存先が正しいか
  - ジョブ数・対象範囲が想定どおりか
```

### ステップ2: dry-run を実行・提示する

`--dry-run`（または相当するオプション）で、**実際には何も変更せず**に計画内容を出力する。
出力には最低限以下を含める。

- 実行する操作の一覧（対象・件数・使う provider やモデル）
- 保存先または送信先
- 推定コスト（分かる場合）

**dry-run の結果をユーザーに提示した時点では本番を実行しない。**

### ステップ3: 明示承認を得る

dry-run の結果をユーザーが確認し、**明示的に「実行してよい」と伝えるまで本番を待つ。**

- 「承認なしに続けて本番まで」進めない。
- 設定変更後の再確認が必要な場合は、変更後に dry-run からやり直す。
- dry-run での承認は、**その provider・設定で実行する承認**のみ。別 provider への自動切替は承認なしに行わない。

### ステップ4: 本番実行と完了確認

承認後に本番を実行する。実行後は、dry-run で示した保存先にファイルが存在することを確認してから「完了」と報告する。

- **API 成功・exit 0 だけで完了扱いにしない。**出力先のファイル・結果が存在することを確認する。
- 保存先を明示して完了報告する（`output-discipline` に従う）。

## 失敗時の対処

HTTP 4xx/5xx などで失敗した場合、**別の手段に自動で切り替えない。**

1. 失敗した操作・エラー内容・影響範囲を報告する。
2. 「待つ」「設定を変える」「範囲を絞る」などの選択肢を提示する。
3. ユーザーの明示指示を待つ。

## 新プロジェクトへの適用

このパターンは次の2点をプロジェクト側で定義すれば動く。

| 定義項目 | 例 |
|----------|-----|
| dry-run の手段 | `--dry-run` フラグ、`--preview` モード、"ジョブ一覧の表示だけ" など |
| 完了確認の手段 | ファイルの存在確認、API レスポンスの検証、DB レコードの確認 など |

それ以外のフロー（設定確認 → dry-run → 承認 → 実行 → 確認）はこのスキルで固定する。

## 禁止

- dry-run の提示前に本番を実行しない。
- ユーザー承認前に「続けて本番まで」進めない。
- 保存先の確認前に「完了した」と言わない。
- 失敗した設定と異なる設定で、承認なしに自動リトライしない。
