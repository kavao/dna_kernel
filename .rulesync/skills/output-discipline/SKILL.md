## 目的

LLM は「チャットに貼れば済む」と判断することがある。このスキルはその抜けを防ぐ。

**完了扱い条件の横断正本は `rules/concepts.md`**。このスキルは、任意のファイル出力作業でその条件を満たすための実行手順を定める。

## 「完了」の定義

次の3ステップを終えた後に限り、ユーザーへ「完了した」と伝えてよい。

1. **ファイルへ書き込む**  
   正本として定めたファイル（テキスト・設定・データ等）に対し、ツールで書き込みを行う。チャットへの貼り付けだけでは完了ではない。

2. **書き込みを確認する**（いずれか）  
   - `Read` で当該ファイルを読み、内容が保存されていることを確認する。追記なら末尾だけでよい。途中挿入なら挿入箇所の前後を確認する。
   - またはファイルサイズ・行数・文字数などを機械的に確認する。

3. **更新パスを添えて報告する**  
   ユーザー向けの完了報告に、更新したファイルのパスを明示する。確認の**前**に「保存した」と言わない。

## バックアップ（上書き更新の前）

既存ファイルを清書・校正・リライトする場合は、上書きの前に旧版を退避する。

```
作業用バックアップの命名例:
  original_file_v001.md
  original_file_v002.md（2回目以降）
```

退避先は `_backup/` や `_old/` など、プロジェクトで統一したフォルダに置く。
退避後に正本ファイルを同一名で更新する。

## 宣言のみで終えない

「これからツールで書き込みます」と述べた場合、**その応答の中**で可能な限り書き込みまで進める。宣言だけを出して区切らない。

- 長くなる場合でも、最低でも書き込みまたは退避のどちらか一歩をツールで実行してから区切る。
- 次の応答では同じ前置きを繰り返さず、未完了ステップから直ちにツール実行で再開する。

**例外**: 外部API呼び出しや課金が伴う操作は、dry-run または計画提示でユーザー承認を待ってから実行する。

## 禁止（幻覚完了の防止）

- ファイルへの書き込みと確認を済ませる前に「完了した」「保存した」と言わない。
- チャットに出力しただけの状態を「ファイルに書いた」と扱わない。

## 新規プロジェクトへの適用

このスキルは次の2点だけ決めれば動く。

1. **正本ファイルの定義**: どのパスのどのファイルが成果物の正本か
2. **確認手段**: `Read` か、プロジェクト固有の検証スクリプトか

それ以外のルール（バックアップ先、命名規則）はプロジェクト側で決める。
