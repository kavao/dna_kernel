## 目的

コードの変更が既存の動作を壊していないことを、テストコードの実行で機械的に確認する。
「動くと思う」ではなく「テストが通った」を完了の根拠にする。

**完了扱いの横断正本は `rules/concepts.md`**（テストによるデグレード防止）。
このスキルはコード変更作業でその条件を満たすための実行手順を定める。

## いつテストを書くか

| 作業 | テストの扱い |
|------|------------|
| 新規スクリプト・ツールの作成 | テストを一緒に書く |
| 既存コードの修正・リファクタリング | 既存テストが通ることを確認。仕様が変わった部分は更新する |
| バグ修正 | バグを再現するテストを書いてから修正する |

## テストの実行パターン

### 変更後の基本確認

プロジェクトで定めたコマンドを実行する（例）:

```bash
python -m pytest tests/
```

```bash
python -m unittest discover
```

終了コード **0** を確認してから完了とする。**0 以外** なら失敗を修正してから再実行する。

### 自己判断での実行タイミング

ユーザーの指示を待たず、次の場面でテストを実行する。

- コードを書き終えた直後
- 他のファイルに影響しうる変更を加えた後
- 完了報告の前

## プロジェクトへの適用

このスキルは次の2点をプロジェクト側で定義すれば動く。

| 定義項目 | 例 |
|----------|-----|
| テスト実行コマンド | `pytest tests/`、`python -m unittest`、`npm test`、`go test ./...` など |
| テストファイルの置き場 | `tests/`、`spec/`、同一ファイル内など |

それ以外の手順（書く → 実行 → 確認 → 完了）はこのスキルで固定する。

## 失敗時の対処

1. 失敗したテストのエラー内容を確認する。
2. 原因を特定し、コードを修正する。
3. テストを再実行して終了コード 0 になることを確認する。
4. 「テストが通った」を添えて完了報告する。

## 禁止

- テストを実行せずにコード変更を「完了」とみなさない。
- テストが失敗した状態のまま「動く」「問題ない」と報告しない。
- テストを黙って削除・コメントアウトして通過させない。
