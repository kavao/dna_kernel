# docs/ 記述ルール

`docs/` 配下のファイルを作成・更新するときは、以下のルールに従う。

---

## 1. 主語の統一

- LLM・アシスタントを指す場合は、**プロジェクトで決めた一つの呼称**（例: 「○○ Coach は」「アシスタントは」）に統一する
- 「私は〜します」は使わない（LLM が約束する一人称表現はユーザー向けドキュメントとして不自然）
- 呼称が決まっていない場合は「アシスタントは」を仮称として使い、決定後に一括置換する

---

## 2. 読者定義

- `docs/` は**システムを使う人間**が読む操作マニュアル
- LLM への行動指示・スキル仕様は書かない（`.rulesync/` の役割）
- `.rulesync/` への参照リンクは「詳細はこちら」程度に留め、ユーザーが直接触る情報を `docs/` に集約する

---

## 3. ページ冒頭の構成

- **入口ページ（index.md）**: 「何ができるか」「誰向けか」を 1〜2 文で先に書く
- **手順ページ**: 「このガイドを読むと〜できる」を冒頭に一文
- **技術詳細ページ**: TL;DR（目的・概要）を冒頭に置き、詳細は後半

---

## 4. 語調・表記

- 丁寧・簡潔（〜します / 〜できます）
- 主語を省略しない
- 箇条書きを活用する

---

## 5. CLI コマンドの記述

コマンドブロックを書くときは、次の順序で構成する。

1. **コマンドの前に**「何をするコマンドか」を 1 文で説明する
2. コードブロックには言語タグを付ける（`bash` / `powershell` など）
3. オプション（`--source`、`--dry-run` など）は日本語でインラインコメントを入れる
4. **dry-run → 本番** の順番で並べる（確認してから実行するフローを体現する）
5. コマンド実行後に「何が起きるか / どこで確認できるか」を一言添える

### 記述例

```
対象フォルダを処理します。
まず --dry-run で対象ファイルと件数を確認してから本番実行します。

# 確認（dry-run）— 対象・件数・保存先を表示する
python tools/my_tool.py target_folder --dry-run

# 本番実行（「OK」を確認してから）
python tools/my_tool.py target_folder

実行後、target_folder/output/ にファイルが保存されます。
```

---

## 6. 内部動作の説明（人の視点）

- **「ユーザーが〜する → アシスタントが〜する → 〜に結果が出る」** の流れで書く
- プロジェクト固有の専門用語は**初出時に一言説明**を付ける
- フローチャート（Mermaid）を使って処理の流れを視覚化する
- 「なぜこうなっているか」の背景を簡潔に触れる
  - 例：「`--dry-run` を先に行うのは、操作が不可逆になる前に内容を確認するためです」
- ユーザーが**見える画面・出力ファイル**を起点に説明する（内部変数・型定義から入らない）

---

## 7. リンク

- 相対パスを使い、自然文の中に埋め込む
- URL の剥き出し表記は避ける
- 誘導先は「詳細」「仕様」ではなく「〜するときはこちら」のように目的で書く

---

## 8. スキル・ルールのドキュメントに「人間の操作視点」を含める

スキルや内部ツール・パイプラインを説明するページには、技術的な仕組みの説明だけでなく、**人間がいつ・どのように使うか**を必ず含める。

ページ内に次の4点をまとめた節（例: 「このドキュメントを使う場面」）を設ける。

1. **どんな場面で使うか** — ユーザーがこのスキルやツールを必要とする状況・前提条件
2. **チャットへの指示文** — 下記「指示文がルール化されている場合の書き方」に従う
3. **アシスタントが行うこと** — 内部で何が起きるかの流れ（箇条書き・順序付き）
4. **ユーザーが確認できるもの** — 生成されるファイル・コンソールに出る結果など

内部の型定義・変数名・アルゴリズムの説明は後半に置き、上記の「操作視点」の節を先に書く。

### 指示文がルール化されている場合の書き方

多くのスキルや動作はルール・スキルとして内部化されており、**短いトリガー文一つで自動的に動く**。指示文の節は次の順序で書く。

1. **最小限のトリガー文を先に示す** — 「これだけ入力すれば動きます」と明示する
2. **「〜と入力すると、アシスタントは自動的に〜します」** の形で内部動作を続けて説明する
3. 詳細なオプション指定が必要な場合のみ、その後に「より詳細に指定したい場合」として補足テンプレートを置く

「コピペ用の長いテンプレート」を冒頭に出すと、ユーザーが「毎回この全文を入力しなければならない」と誤解する。**実際はシンプルな一言で動く仕組みになっていること**を必ず先に明示する。

---

## 9. 新しいスキルを追加したときの docs 反映

`.rulesync/skills/<skill_name>/SKILL.md` を新しく追加したときは、ユーザーがそのスキルの存在と成果物を見つけられるように、`docs/` 側も確認する。

- 既存ページで十分な場合は、そのページから新スキルへリンクを追加する。
- 独立して説明した方がよい場合は、`docs/` 配下にユーザー向けページを追加する。
- 関連する索引ページ（例: `docs/index.md`）へリンクを追加する。
- docs を追加しない判断をする場合は、理由を査証ログに残す。

docs にはスキルの内部仕様を丸写しせず、ユーザーが入力する指示文、アシスタントが行うこと、保存されるファイルを中心に書く。
