# 自己発展型ルールガバナンス

## これは何か

LLM ハーネスを使うプロジェクトで、
**ルールが肥大・矛盾せず**、**完了が曖昧にならず**、**履歴が改ざんされない**ようにする運用パターンです。

「チャットで言い続ければ守られる」という運用の代わりに、
3つの構造で LLM の行動を拘束します。

## 3つの構造

### 1. 概念正本（Policy-as-Code）

`rules/concepts.md` に「短い定義・禁止・完了条件」だけを置きます。
スキルや他のルールは、その概念へのリンクと「この作業での適用」だけを書きます。

**効果**: 矛盾の発生源を1か所に集め、変更の影響範囲を管理できる。

### 2. 完了の拘束

「完了」は、成果物がファイルに実在し、再読込または機械検証で確認が済んだ後だけと定義します。
LLM が「書きました」と言っても、ファイルへの書き込みと確認を終えるまでは完了扱いにさせません。

**効果**: 「会話に出したが保存していない」「dry-run 前に本番を走らせた」を防ぐ。

### 3. 追記専用の査証ログ

作業後に `_workingspace/log/YYYYMM.md` へ追記します。上書きと削除は禁止です。
`workspace_audit_log.py` を使えば、既存行を変更できないコマンドで追記できます。

**効果**: 「何をしたか」の事実が残り、「やった・やらない」の争いが起きない。

## ルールが自己発展する仕組み

1. 新しい作業パターンが出てきたら、`concepts.md` へ短い定義を追加する
2. 既存スキルや指示は、その定義へのリンクに置き換える（本文を複製しない）
3. 追加した事実を査証ログへ残す

これを繰り返すことで、ルールセットが「増やすほど薄くなる」構造になります。
概念本文は1か所に集まり、各スキルはその概念を参照するだけになっていくためです。

## 最小カーネル（移植に必要なもの）

| 要素 | ファイル | 役割 |
|------|----------|------|
| 概念正本 | `rules/concepts.md` | 定義・禁止・完了条件の一本化 |
| ルール作法 | `rules/rule-authoring.md` | 増やし方の固定 |
| 査証ログ | `tools/kernel/workspace_audit_log.py` | 追記専用の作業記録 |
| 機械検証 | `tools/kernel/novel_project_check.py` | 完了条件の自動確認 |

**最小構成**: 概念正本 + 査証ログツールだけでも効果があります。
機械検証ツールは「完了の拘束」を強化するオプションです。

## LLM 向けの入口に書くこと

`CLAUDE.md` / `AGENTS.md` に以下の2行を加えれば動きます。

```
- ルールの正本は `rules/concepts.md`（または移植先での相当パス）。変更はここを先に更新する。
- 作業後は `tools/kernel/workspace_audit_log.py` で査証ログへ追記する。
```

あとは `concepts.md` の「完了扱い条件」と「査証ログの原則」に従って動きます。
